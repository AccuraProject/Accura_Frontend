¿Qué es un prompt para generar una regla?

Un prompt es la instrucción que el usuario escribe para solicitar a la IA la creación de una regla de validación.

Mientras más claro y específico sea el prompt, mejor será la regla generada por la IA.

Estructura recomendada

Para crear una regla de validación, se recomienda redactar el prompt con la siguiente estructura:

Validar que el campo "[Nombre del campo]" sea de tipo [Tipo de dato] y cumpla con [Condición]; en caso contrario, mostrar el mensaje: "[Mensaje de error]".

Ejemplo válido: [Valor válido].

Ejemplo inválido: [Valor inválido].

Se utiliza cuando el campo debe contener texto, como nombres, apellidos, códigos o descripciones.

Validar que el campo "Nombre" sea de tipo texto y contenga solo letras; en caso contrario, mostrar el mensaje: "El nombre solo debe contener letras".

Ejemplo válido: Carlos.

Ejemplo inválido: Carlos123.

 Regla de tipo Número

Se utiliza cuando el campo debe contener valores numéricos, como montos, porcentajes, edades, cantidades, tasas o importes.

Validar que el campo "Porcentaje" sea de tipo número y tenga un valor entre 0 y 100; en caso contrario, mostrar el mensaje: "El porcentaje debe estar entre 0 y 100".

Ejemplo válido: 80.

Ejemplo inválido: 150.

 Regla de tipo Fecha

Se utiliza cuando el campo debe contener una fecha con un formato específico.

Validar que el campo "Fecha de nacimiento" sea de tipo fecha y tenga el formato DD/MM/AAAA; en caso contrario, mostrar el mensaje: "La fecha de nacimiento debe tener el formato DD/MM/AAAA".

Ejemplo válido: 15/08/1995.

Ejemplo inválido: 1995-08-15.

 Regla de tipo Correo

Se utiliza para validar direcciones de correo electrónico.

Validar que el campo "Correo electrónico" sea de tipo correo y tenga un formato válido; en caso contrario, mostrar el mensaje: "El correo electrónico no tiene un formato válido".

Ejemplo válido: usuario@empresa.com.

Ejemplo inválido: usuarioempresa.com.

 Regla de tipo Teléfono

Se utiliza para validar números telefónicos.

Validar que el campo "Teléfono" sea de tipo teléfono y tenga exactamente 9 dígitos numéricos; en caso contrario, mostrar el mensaje: "El teléfono debe tener exactamente 9 dígitos".

Ejemplo válido: 987654321.

Ejemplo inválido: 98765ABC.

 Regla de tipo Documento

Se utiliza para validar documentos de identidad o identificación tributaria, como DNI, RUC, Carnet de Extranjería o Pasaporte.

Validar que el campo "Número de documento" sea de tipo documento y tenga exactamente 8 dígitos numéricos; en caso contrario, mostrar el mensaje: "El número de documento debe tener exactamente 8 dígitos".

Ejemplo válido: 12345678.

Ejemplo inválido: ABC12345.

 Regla de tipo Lista

Se utiliza cuando un campo solo puede aceptar ciertos valores definidos.

Validar que el campo "[Nombre del campo]" sea de tipo lista y solo permita los valores: [Valor 1], [Valor 2], [Valor 3]; en caso contrario, mostrar el mensaje: "[Mensaje de error]".

Ejemplo válido: [Valor permitido].

Ejemplo inválido: [Valor no permitido].

Validar que el campo "Estado de póliza" sea de tipo lista y solo permita los valores: Activa, Anulada, Vencida; en caso contrario, mostrar el mensaje: "El estado de póliza ingresado no es válido".

Ejemplo válido: Activa.

Ejemplo inválido: Pendiente.

 Regla de tipo Lista compleja

Se utiliza cuando la validación no depende de un solo campo, sino de la combinación válida entre varios campos relacionados.

Este tipo de regla permite verificar que una combinación de columnas exista o sea correcta dentro de un catálogo definido. Por ejemplo, para los campos Departamento, Provincia y Distrito, no basta con que cada valor exista por separado; también debe validarse que la relación entre ellos sea correcta.

Validar que la lista compleja de los campos "[Campo 1]", "[Campo 2]" y "[Campo 3]" sea válida según [Catálogo o referencia]; en caso contrario, mostrar el mensaje: "[Mensaje de error]".

Ejemplo válido: [Campo 1]: [Valor], [Campo 2]: [Valor], [Campo 3]: [Valor].

Ejemplo inválido: [Campo 1]: [Valor], [Campo 2]: [Valor], [Campo 3]: [Valor].

Validar que la lista compleja de los campos "Departamento", "Provincia" y "Distrito" sea válida según el catálogo de ubicación geográfica; en caso contrario, mostrar el mensaje: "La combinación de departamento, provincia y distrito no es válida".

Ejemplo válido: Departamento: Lima, Provincia: Lima, Distrito: Lima.

Ejemplo inválido: Departamento: Áncash, Provincia: Lima, Distrito: San Miguel.

Regla de tipo Dependencia

Una regla de dependencia se utiliza cuando la validación de un campo cambia según el valor de otro campo.

A diferencia de la lista compleja, en una dependencia un campo condiciona la regla que debe cumplir otro campo.

Validar que el campo "[Campo dependiente]" dependa del campo "[Campo condicionante]".

Si "[Campo condicionante]" es "[Valor]", entonces "[Campo dependiente]" debe cumplir con [Condición].

En caso contrario, mostrar el mensaje: "[Mensaje de error]".

Ejemplo válido: [Combinación válida].

Ejemplo inválido: [Combinación inválida].

Validar que el campo "Número de documento" dependa del campo "Tipo de documento".

Si "Tipo de documento" es "DNI", entonces "Número de documento" debe tener exactamente 8 dígitos numéricos.

Si "Tipo de documento" es "RUC", entonces "Número de documento" debe tener exactamente 11 dígitos numéricos.

En caso contrario, mostrar el mensaje: "El número de documento no cumple con el formato requerido según el tipo de documento" .

Ejemplo válido: Tipo de documento: DNI, Número de documento: 12345678.

Ejemplo inválido: Tipo de documento: DNI, Número de documento: 1234.
